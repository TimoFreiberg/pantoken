### Goal

Make remote Pantoken connections feel like a capability of the released `Pantoken.app`: the user supplies a working SSH destination, and Pantoken automatically provisions the matching remote Pantoken runtime, verifies the host's polytoken compatibility, optionally installs polytoken into the remote Pantoken state area, and connects through a persistent lightweight remote runtime without requiring a manually configured service or dedicated remote TCP port.

The existing Pantoken logical message contract and Svelte UI semantics remain stable. WebSocket remains the local/browser-facing transport; remote desktop/mobile connections use an SSH-launched stdio transport and a local bridge/adaptor rather than tunneling WebSocket itself.

### Implementation Summary

Introduce a transport-neutral, framed Pantoken wire layer beneath the existing JSON `ClientMessage`/`ServerMessage` contract. The Rust server's current Axum WebSocket connection pump becomes an adapter over a reusable bidirectional message session; a new stdio adapter uses length-prefixed JSON frames over SSH stdin/stdout with `-T`, while stderr carries diagnostics only.

Add a per-user remote runtime, installed under `~/.local/share/pantoken` by default (with an explicit override for tests/advanced use). The runtime consists of versioned Pantoken artifacts with the currently available checksum-only provenance (matching the existing standalone headless deployment model; a future signing mechanism may strengthen this) plus a short-lived SSH stdio proxy and a persistent lightweight server/hub process. The proxy reconnects to the persistent server through a private Unix socket; the server owns durable session metadata and polytoken daemon lifecycles. No public remote Pantoken port is required.

Do not bundle polytoken into the Pantoken release by default. During provisioning, probe `polytoken --version` from the remote PATH and compare it against a generated, embedded API-target version extracted from the `polytoken openapi` spec used to generate `server-rs/pantoken-daemon-types`. A missing or too-old polytoken is an explicit compatibility state: the user may install/repair it from the official release artifacts documented at https://docs.polytoken.dev/installation/downloads/#release-artifacts, or configure/provide an existing executable. Downloaded polytoken binaries live under the remote Pantoken root, are selected by remote OS/architecture, and are verified against the published SHA256 manifest. The plan must not claim signature verification for polytoken until its upstream distribution publishes a verifiable signature mechanism; Pantoken should surface checksum-only provenance honestly.

Primary touch points are `server-rs/pantoken-server/src/main.rs`, a new/reworked transport/codec module near `server-rs/pantoken-server`, `server-rs/pantoken-protocol/src/wire.rs`, `client/src/lib/ws.svelte.ts`, `desktop/src/supervisor.rs`, `desktop/src/config.rs`, the desktop native connection/IPC layer, `server-rs/pantoken-server/src/polytoken/daemon_client.rs`, `scripts/codegen-polytoken-rs.ts`, and remote release/deployment tooling. `protocol/src/wire.ts` is the logical-contract reference and should not change unless the bridge requires new message types. Existing local WebSocket behavior and local bundled-server supervision are out of scope for removal.

### Implementation Plan

#### Phase 0 — Freeze the remote contracts and release metadata

1. Define a transport-neutral `WireMessage`/connection-session contract around the existing `ClientMessage` and `ServerMessage` types. The existing enums and `PROTOCOL_VERSION` in `server-rs/pantoken-protocol/src/wire.rs` remain unchanged; the transport-neutral contract is a new wrapper/session type that composes them. Preserve the current protocol version, hello/resume semantics, `buildSha`, seeds, event sequence/epoch handling, ping/pong, and error behavior. Document that WebSocket and stdio are adapters, not separate application protocols.
2. Define the stdio framing format: a bounded unsigned length prefix (use one canonical byte order and integer width), followed by one UTF-8 JSON envelope; reject oversized frames, malformed lengths, invalid UTF-8, and truncated EOFs deterministically. Make stdout protocol-only and stderr log-only.
3. Before implementation handoff, inspect a real `polytoken openapi` output and verify the authoritative API-version field; do not assume `info.version` because the current generator interface does not capture `info` and no OpenAPI fixture is committed. Make this a Phase 0 gate: if no parseable authoritative version exists, stop and resolve the upstream/version-source decision before implementing compatibility checks. Once verified, extend `scripts/codegen-polytoken-rs.ts` to extract that field, validate it as a supported semver/prerelease string, and emit a generated constant in `pantoken-daemon-types` such as `POLYTOKEN_OPENAPI_TARGET_VERSION`. Add a generated-source comment, a committed fixture or deterministic generator-input seam, and a test/CI check that the constant changes when the source spec version changes.
4. Add a Pantoken remote release manifest contract containing Pantoken release/build identity, protocol version, supported target triples, artifact URLs, SHA256 digests, and any required runtime capabilities. Keep the Pantoken helper artifact identity separate from the polytoken compatibility floor: Pantoken helper exact release/build matching is required; polytoken is a minimum API-compatible version check with an explicit newer-version policy.
5. Specify remote layout under the default root `~/.local/share/pantoken`, for example:
   - `releases/<pantoken-version>/<target>/pantoken-server` for immutable Pantoken artifacts;
   - `tools/polytoken/<polytoken-version>/<target>/polytoken` for optional downloaded polytoken binaries;
   - `run/` for private socket, pid/lock, and runtime metadata;
   - `logs/` for bounded diagnostic logs;
   - `install.json`/manifest metadata for installed versions and provenance.
   Keep the existing local server's `XDG_STATE_HOME` default unchanged in this effort unless an explicit follow-up migration is approved; the requested `~/.local/share/pantoken` default applies to the remote runtime/provisioning root.

#### Phase 1 — Stdio protocol and persistent remote runtime

6. Substantially replace (rather than thread transport conditionals through) the server-side WebSocket handling in `server-rs/pantoken-server/src/main.rs` with a reusable connection session in a concrete module such as `server-rs/pantoken-server/src/connection/mod.rs` (or an equivalently named crate-root module). The Axum handler remains a thin WebSocket adapter. The reusable session owns: hello authentication/registration, hub input dispatch, outbound message pumping, orderly close, heartbeat, and removal from the hub.
7. Add a stdio adapter for the same connection session. It must run without a PTY, use the framed codec, keep stdout free of logs, preserve backpressure, handle half-close/EOF, and map transport failures to the existing reconnect/resume behavior. Add a remote-proxy command mode that accepts the framing on stdin/stdout and connects to the persistent runtime's private Unix socket. The proxy must race-safely bootstrap the persistent server: try the private socket first, acquire the runtime lock if absent, spawn the versioned server with the resolved polytoken/XDG configuration, wait for socket readiness with a bounded timeout, then connect; concurrent proxies must converge on one server and stale pid/socket metadata must be recoverable.
8. Add a persistent remote server/runtime mode. It should listen only on a user-owned Unix socket below the remote root, enforce single-instance locking, write bounded logs, and expose enough health/identity metadata for the proxy to distinguish “not installed,” “starting,” “incompatible,” and “running.” The runtime must not bind a public TCP listener for normal SSH connections.
9. Build the remote lifecycle manager as new infrastructure; the current `idle_reap_ms` setting is not wired to daemon cleanup. Define process lifetime precisely:
   - track client connections, active turns, warm attachments, daemon PIDs, and process start tokens explicitly;
   - an active turn or active client connection prevents cleanup;
   - a dropped SSH proxy does not terminate a running polytoken turn or delete durable session data;
   - when there are no connections and no active turns, a configurable grace-period task may dispose warm session attachments/stop idle per-session polytoken daemons while retaining resumable session state;
   - when the hub itself has no connections, no active turns, and no warm work, it may exit after an idle grace period; the next proxy invocation must race-safely start it again;
   - cleanup must use pid/start-token identity checks and never kill an unrelated process;
   - add explicit lifecycle tests before relying on AC.10.
10. Implement explicit reconnect: new SSH process/proxy, reconnect to the persistent runtime, hello with the existing resume token, tail replay when covered, and full seed fallback otherwise. Add bounded backoff, stale-socket recovery, startup timeouts, and diagnostics for remote process exit versus network/auth failure.
11. Provide a local bridge/adaptor so the unchanged WebView client can continue to connect to a loopback WebSocket while the native layer forwards messages to the SSH stdio transport. Specify its lifecycle and ownership: bind an app-owned loopback port using the existing free-port approach, accept the browser's existing hello/resume/heartbeat protocol, forward JSON messages into the framed SSH stream without changing logical envelopes, and map SSH EOF/exit/timeout into browser close/reconnect behavior. A browser reconnect must create a fresh SSH proxy while preserving the resume token; the bridge must not duplicate hello, swallow ping/pong, or leak credentials. Keep this bridge behind a transport abstraction that can later be used by mobile native code. Add a bridge forwarding/error-propagation integration test.

#### Phase 2 — Desktop/mobile connection UX and SSH lifecycle

12. Add a persisted remote-host model/profile with SSH destination, username, port/config alias, known host label, preferred polytoken policy (`require-existing` versus `offer-install`), and optional runtime-root override. Do not persist plaintext SSH passwords or private keys.
13. Implement SSH lifecycle ownership in the desktop native layer: use the platform SSH client/config semantics initially, invoke commands with `-T` and explicit exit/error handling, surface host-key and key-passphrase prompts, and keep all sensitive command arguments/log output redacted. Add keepalive and bounded reconnect policy; do not use reverse forwarding or a public remote port in the initial design.
14. Add a native desktop/mobile transport test seam that injects a fake SSH transport, lets tests drive each provisioning/connection phase deterministically, and keeps the WebView scenario tests independent of a real SSH process.
15. Add connection-state UX with phase-specific progress: testing SSH, inspecting target, checking Pantoken runtime, checking polytoken, downloading, uploading, verifying, installing, starting, reconnecting, and ready. Distinguish actionable failures: SSH authentication/host-key, unsupported OS/architecture, no writable user directory, missing/too-old polytoken, checksum mismatch, Pantoken artifact mismatch, runtime startup failure, and protocol mismatch.
16. Add the corresponding native transport abstraction for mobile. The mobile UI should use the same logical connection/provisioning state model, but the platform implementation may use a native SSH library rather than a system executable. Account for app suspension by reconnecting on foreground and relying on remote persistence for active turns.
17. Ensure remote identity is represented in the existing hello/server-label model and settings UI so filesystem paths, logs, sessions, and current host are unambiguous. Avoid treating a remote connection as merely a different port.

#### Phase 3 — Remote provisioning and polytoken compatibility/install

18. Implement a dependency-light remote probe executed through SSH. It should report OS, architecture, bitness, relevant libc/runtime information, home directory, writable/executable locations, available `mkdir`/`chmod`/`mv`/`sha256`/archive tools, and the result of `polytoken --version` without depending on shell startup noise. Parse the last structured probe record rather than trusting arbitrary login-shell output.
19. Resolve the Pantoken helper artifact from the release manifest using the probed target and exact app release/build identity. Prefer downloading on the local device and uploading over SSH/SFTP or a raw SSH transfer channel so remote outbound HTTPS is not required. If a remote-download mode is retained, make it explicit and diagnostically separate.
20. Verify Pantoken artifacts before execution: HTTPS/download integrity, manifest SHA256 digest, remote post-transfer digest, executable bit, and an atomic rename into the versioned install directory. The current standalone Pantoken deployment has checksum/archive validation but no defined signature-verification mechanism; do not claim signature verification unless a concrete release signing scheme, public-key distribution, and verifier are added as separate scope. Never replace a currently working version in place. Retain at least one previous helper version for rollback and garbage-collect only unreferenced versions.
21. Probe polytoken independently from the helper. Search the configured remote PATH and any previously recorded path, run `polytoken --version`, parse stable and prerelease semver, and compare against generated `POLYTOKEN_OPENAPI_TARGET_VERSION`. Policy:
   - missing: show “Polytoken is required” with an install action when the profile allows it;
   - lower than the generated target: warn/block normal connection according to the API compatibility policy, with a clear upgrade/install action;
   - equal/newer: accept, while recording the observed version and any newer-than-target notice;
   - unparseable or command failure: treat as an actionable compatibility failure, not as success.
22. Add the optional polytoken installer using the official artifact matrix (`linux`/`macos` × `amd64`/`arm64`, `tar.gz` or `zip`) and checksum manifest at the documented release URL. Resolve a configured target release/channel rather than silently picking an arbitrary latest unstable build. Download locally, verify SHA256 against the appropriate published manifest, extract safely, install to `tools/polytoken/<version>/<target>/polytoken`, and write provenance metadata. Because the current upstream page documents checksums but no signatures, label the trust level accordingly and make the release URL/channel visible in the UI.
23. Configure the remote Pantoken runtime to use the selected existing or downloaded polytoken path without overwriting the user's PATH or global polytoken installation. Persist the resolved binary path in the remote runtime's install metadata and pass it explicitly to the server as `PANTOKEN_POLYTOKEN_BIN` (or an equivalent typed runtime configuration) when the proxy starts the persistent server. Define the XDG ownership contract: when polytoken is Pantoken-managed (downloaded/installed under the remote root), isolated `XDG_CONFIG_HOME`/`XDG_DATA_HOME`/`XDG_CACHE_HOME` roots below the remote Pantoken root are mandatory. When polytoken is user-provided/existing, sharing the user's existing polytoken XDG roots requires explicit user-facing confirmation and a persisted profile choice; Pantoken must never silently redirect an existing installation or silently share production state. In both modes document where sessions, config, credentials, caches, and logs live and test the isolation boundary.
24. Add idempotent provisioning reconciliation: reconnecting to an already-provisioned host should only run lightweight probes and version checks; interrupted uploads/installations must leave no executable at the final path until verification succeeds; failed upgrades must preserve the last-known-good helper and polytoken binary.

#### Phase 4 — Integration, test harness, and rollout

25. Build deterministic test infrastructure because the current repository has protocol/unit/e2e coverage but no remote-host bootstrap harness or native SSH injection seam. The SSH/provisioning harness must be explicit and composable:
   - a fake SSH process/transport emulator covering command execution, stdin/stdout framing, stderr diagnostics, exit codes, disconnects, and reconnects;
   - an isolated remote filesystem model covering mkdir/chmod/mv/atomic install/hash/archive extraction and permissions;
   - injectable probe responses for OS/architecture/libc/tools/polytoken versions;
   - failure injection for transfer, checksum, extraction, process startup, stale sockets, and concurrent startup;
   - persistent fake runtime state so active turns and idle GC can be tested.
   The Phase 2 native desktop/mobile transport seam should define its injectable interface before Phase 4; its smoke tests (`native_transport_seam_smoke_tests`) are written in Phase 4 after the fake transport harness exists. The harness and seam must remain decoupled from real SSH processes. Keep all harnesses hermetic and never use the operator's real SSH config, home, polytoken sessions, or production state.
26. Add an end-to-end local integration scenario that starts a fake remote runtime through the SSH adapter, provisions a target-specific Pantoken artifact, exercises hello/seed/prompt/event/ping/resume, drops the proxy, reconnects, and confirms that an active session survives while an idle session is eventually reaped. Add variants for existing compatible polytoken, missing polytoken with install accepted, missing polytoken with install declined, too-old polytoken, unsupported target, checksum mismatch, and interrupted installation.
27. Add release/build checks for all declared remote target artifacts and the generated OpenAPI target version. Add a manifest validation test that rejects missing target entries, digest mismatches, duplicate identities, or an artifact whose embedded build metadata does not match its manifest.
28. Run the existing protocol, Rust, client, desktop, and e2e gates, then perform manual validation against a disposable localhost/container/VM SSH host for host-key prompts, a clean home directory, no remote outbound HTTPS, existing/new/old polytoken cases, reconnect during an active turn, and cleanup after idle. Do not use the production remote host as the first validation target.

### Acceptance Criteria

- **AC.1 — Transport parity:** The same `ClientMessage`/`ServerMessage` semantics, protocol-version guard, hello/resume flow, ping/pong, event sequencing, and seed fallback work over both the existing WebSocket adapter and framed SSH stdio. → `transport_codec_roundtrip_tests`, `websocket_adapter_contract_tests`, `stdio_adapter_contract_tests`, `remote_hello_resume_integration`.
- **AC.2 — Safe framing:** Stdio rejects malformed/oversized/truncated frames, preserves message boundaries under partial reads/writes, and never emits logs on stdout. → `frame_codec_rejects_invalid_input`, `frame_codec_handles_fragmentation`, `stdio_stdout_is_protocol_only`.
- **AC.3 — No dedicated remote port:** A provisioned remote connection reaches the runtime using only the user's SSH destination and a remote exec/stdin/stdout proxy; no public Pantoken TCP listener is required. → `ssh_stdio_no_remote_port_integration` plus manual disposable-SSH validation.
- **AC.4 — Seamless Pantoken provisioning:** Given a working SSH account and a supported target, Pantoken selects the matching release artifact, transfers it, verifies it, installs it atomically below the remote root, starts the proxy/runtime, and reaches a ready UI without manual remote service setup. → `provision_clean_host_e2e`.
- **AC.5 — Remote root:** The default remote runtime/provisioning root is `~/.local/share/pantoken`; all runtime paths are derived safely from it, and an explicit test override is supported. → `remote_root_default_and_override_tests`.
- **AC.6 — Generated compatibility target:** The generated daemon-types crate exports the OpenAPI source version used for code generation, and every remote server binary exposes that target version in its identity/diagnostic handshake. → `codegen_embeds_openapi_version`, `remote_identity_reports_api_target`, generated-source CI check.
- **AC.7 — Polytoken policy:** Existing compatible polytoken is accepted; missing polytoken, unparseable version, and lower-than-target version produce distinct actionable states; the user can choose to require an existing installation or install the selected official release artifact. → `polytoken_compatibility_matrix`.
- **AC.8 — Optional polytoken installation:** When installation is accepted, Pantoken selects the correct documented OS/architecture artifact, verifies the published SHA256 manifest, installs it under the remote Pantoken root without modifying the user's global installation, and records provenance. → `polytoken_install_artifact_matrix`, `polytoken_checksum_failure_is_atomic`.
- **AC.9 — Persistent runtime/reconnect:** A dropped SSH connection does not kill an active polytoken turn; a new proxy reconnects to the persistent runtime and resumes via tail or full seed; stale sockets/processes are recovered safely. → `active_turn_survives_proxy_drop`, `resume_tail_or_seed_fallback`, `stale_runtime_recovery`.
- **AC.10 — Lightweight cleanup:** When there are no connections and no active turns, idle warm session daemons are reaped after the configured grace period while durable session state remains resumable; the hub may exit when fully idle; active turns are never reaped. → `idle_session_gc_preserves_history`, `active_session_not_reaped`, `idle_hub_shutdown_and_restart`.
- **AC.11 — UX diagnostics:** Desktop/mobile connection UX exposes phase-specific progress and distinguishes SSH, target, artifact, polytoken, runtime, and protocol failures without leaking credentials. → `connection_state_render_scenarios` plus manual host-key/passphrase validation.
- **AC.12 — Regression safety:** Existing local WebSocket, bundled desktop server, mock/fake driver, protocol, client, and e2e behavior remain green. These are existing regression gates; new remote behavior is covered by AC.1–AC.11 and AC.13. → `bun test`, `bun run check`, `bun run check:rs`, `bun run test:e2e`, `cargo fmt --check -p pantoken-desktop`, `cargo clippy --locked -p pantoken-desktop --all-targets -- -D warnings`, and `cargo test -p pantoken-desktop`. A standalone `cargo check --manifest-path desktop/Cargo.toml` may be used as a supplementary local loop.
- **AC.13 — Test infrastructure:** The fake SSH/remote-filesystem harness and native transport seam can deterministically drive framing, probe, transfer failure, persistent-runtime restart, and connection-phase scenarios without touching real SSH or user state. → `fake_ssh_harness_smoke_tests`, `native_transport_seam_smoke_tests`.

### Test Strategy

- **Unit tests:** framing codec; JSON envelope round-trips; frame limits; semver/prerelease parsing and minimum-version comparison; target normalization; default-root/path safety; manifest identity/digest validation; probe-output parsing; install-state reconciliation; PID/start-token cleanup decisions.
- **Rust integration tests:** reusable connection-session behavior independent of transport; WebSocket and stdio adapters against the same scripted hub; Unix-socket proxy/runtime lifecycle; hello/resume/tail replay; active-turn survival; idle GC; stale process/socket handling.
- **Provisioning integration tests:** the new isolated fake-SSH/remote-filesystem harness exercises clean install, already-installed reuse, interrupted transfer, checksum mismatch, atomic rollback, supported/unsupported targets, existing compatible polytoken, missing polytoken, old polytoken, and optional installation from a local fixture of the documented artifact/manifest shape.
- **Client/native scenario tests:** connection phase rendering, retry/cancel behavior, host identity, compatibility warning/install choice, and error redaction. Use the existing Playwright e2e layer for observable WebView behavior and a native test seam/fake transport for desktop/mobile lifecycle behavior.
- **Release checks:** generated OpenAPI version is present and non-empty; `openapi_version_ci_gate` runs `polytoken openapi`, extracts the verified authoritative version field, and asserts it equals the generated `POLYTOKEN_OPENAPI_TARGET_VERSION` constant, failing on divergence; the remote manifest target matrix is complete; each artifact's SHA256 metadata is internally consistent; the remote helper reports the release/build identity expected by the manifest. Any future signature metadata is out of scope until a concrete signing scheme is defined.
- **Manual validation:** disposable localhost/VM/container SSH server with a clean home; desktop Rust validation must include `cargo check --manifest-path desktop/Cargo.toml` and `cargo clippy --manifest-path desktop/Cargo.toml --all-targets -- -D warnings`; host-key confirmation and key-passphrase prompt; no remote outbound HTTPS; real target artifact install; existing/old/missing polytoken; active-turn disconnect/reconnect; idle cleanup and durable resume. Manual checks are supplementary, not replacements for the hermetic harness.
- **Test infrastructure acceptance:** the fake SSH/remote-filesystem harness and native transport seam must each have smoke tests that prove framing, probe injection, artifact failure injection, phase driving, and persistent-runtime restart behavior before higher-level provisioning/UX tests rely on them.

### Risks, Blockers, and Required Decisions

- **Upstream polytoken trust:** the official download page currently documents SHA256 manifests but no signature files or verification procedure. The installer must expose checksum-only provenance unless upstream adds signatures; do not imply stronger provenance for either polytoken or Pantoken standalone helper artifacts than the release process actually provides.
- **Pre-implementation blocker — generated version source:** this is a Phase 0 gate, not an implementation-time assumption. The handoff must first inspect a real `polytoken openapi` output and record the authoritative version field and parseable format; if no such field exists, stop and resolve the upstream/version-source decision before implementing compatibility checks. Inspect a real `polytoken openapi` output before handoff; if it does not expose a parseable authoritative version, stop and resolve the upstream/version-source decision before implementing compatibility checks. The code generator must fail loudly rather than emitting an unversioned compatibility floor.
- **Version semantics:** a newer polytoken may still remove/change behavior despite being numerically newer. Start with the requested minimum-version warning policy, but surface newer-than-target status and keep the live corpus/contract tests as the compatibility gate.
- **Platform matrix:** Pantoken remote artifacts must exist for every advertised target; unsupported OS/architecture must fail before transfer. Initial scope should prioritize macOS amd64/arm64 and Linux amd64/arm64, matching the current polytoken release artifact matrix.
- **Runtime data ownership:** the plan installs remote Pantoken and optional polytoken binaries under `~/.local/share/pantoken`, but polytoken's own XDG data/config conventions must be preserved or explicitly isolated to avoid touching an unrelated global installation. Resolve the exact environment layout in implementation and test it with an isolated home.
- **Mobile SSH implementation:** arbitrary SSH cannot be initiated by a browser/PWA alone. Mobile requires a native SSH transport implementation; the shared transport/provisioning contract should be built first, while mobile native integration may be a separately sequenced execution task.
- **SSH command-shell variability:** remote login shell startup files can emit noise or alter PATH. The probe and proxy commands must use a controlled non-interactive invocation and parse structured records; diagnostics must tell the user when shell initialization is the cause.
- **Security boundary:** SSH host-key verification is the primary host identity; the remote Unix socket must be user-private; no agent forwarding or reverse forwarding should be enabled by default; cleanup must verify process identity before signaling.
